# Configs

Match configs use `schemaVersion` `fps-arena-bench.schema.v0.1` and should pin map version/hash, seed, contenders, ruleset version, timeout budgets, fallback policy, and debug capture settings.

Required fields are `schemaVersion`, `id`, `rulesetVersion`, `map`, `seed`, `maxTicks`, `contenders`, `actionTimeoutMs`, `invalidActionPolicy`, and `capture`.

`id`, `rulesetVersion`, `map.id`, and `map.version` must be non-empty strings. `map` contains `id`, `version`, and a `sha256:<64 lowercase hex digest>` hash for the map file. `seed` must be a nonnegative integer; `maxTicks` and `actionTimeoutMs` must be positive integers. `contenders` must include at least two unique non-empty `id` strings and required non-empty `adapterId` strings, with optional non-empty `displayName`; multiple contenders may reuse the same `adapterId`. `invalidActionPolicy` contains nonnegative integer `maxInvalidActions` and `fallbackAction`, where fallback is either `noop` or `repeat-last-valid`. Provider errors, timeouts, and invalid action schemas use that fallback; `repeat-last-valid` reuses the contender's previous valid action and falls back to `noop` when none exists. CLI runs record invalid-action counters but do not currently terminate on `maxInvalidActions`; the field is reserved for invalid-action termination. Persisted results report these as `schemaFailures`, `timeouts`, and total `fallbackActions`; CLI summaries also expose provider-error and schema-violation counts. `capture` contains `safeReplay` and `privateDebug` booleans; CLI runs currently always write `replay.safe.json` and `result.json` and do not emit `debug.private.jsonl`. Runtime-only local adapter settings, such as Ollama model/endpoint or Claude/Codex/OpenCode CLI enablement, command, and timeout overrides, are supplied outside the strict match config through provider factory overrides or CLI environment variables.

## Batch configs

Batch configs use the dedicated `schemaVersion` `fps-arena-bench.batch.v0.2` and drive the `fps-arena-bench batch` command across a deterministic matrix of seeds, maps, contender matchups, and spawn permutations. They reuse match-level `rulesetVersion`, `actionTimeoutMs`, `maxTicks`, `invalidActionPolicy`, and `capture` blocks so per-match config artifacts written under each batch output dir validate against the same `MatchConfigSchema` used by the single-match runner.

Required fields: `schemaVersion`, `id`, `rulesetVersion`, `seeds`, `maps`, `contenders`, `matchups`, `spawnPermutations`, `maxTicks`, `actionTimeoutMs`, `invalidActionPolicy`, `capture`, and `failurePolicy`. Optional: `runLimits.maxMatches`. `seeds` is a non-empty array of unique nonnegative integers. `maps` is a non-empty array of `{id, version, path}` entries with unique `id` values; `path` resolves relative to the batch config file and must point to a `MapSchema`-valid file whose `id` and `version` match the entry. The runner computes `sha256:<digest>` for each referenced map at run time so per-match configs always match the loaded map. `contenders` is a non-empty pool of `{id, adapterId, displayName?}` with unique `id` values. `matchups` is a non-empty array of `{id, contenderIds}` entries; `contenderIds` references contender ids from the pool, must be unique within a matchup, and every matchup must have the same number of contenders. `spawnPermutations` is a non-empty array of permutations whose length matches the matchup size, where each entry is a unique slot index in `[0, size)` and slot `i` of a permutation selects which matchup contender plays spawn slot `i`. `failurePolicy.onMatchFailure` is `continue` (record the failure and keep running) or `stop` (mark remaining planned matches as `skipped`). `runLimits.maxMatches` truncates the deterministically planned schedule to the first `n` matches when set.

Batch identifiers and per-match identifiers are constrained to ASCII letters, digits, underscore, hyphen, and period because they are used in match ids and on-disk path components. Match ids are deterministically generated as `<batchId>__<mapId>__<matchupId>__p<permIndex>__s<seed>`. The runner iterates maps × matchups × spawnPermutations × seeds in declared order, so the same config produces the same plan and manifest on every run.

The `batch` subcommand writes `<outDir>/<batchId>/manifest.json` plus `<outDir>/<batchId>/matches/<matchId>/{config.json,replay.safe.json,result.json}` for every completed match. `manifest.json` records the batch config snapshot, ordered runs, per-run status (`completed`, `failed`, or `skipped`), winner/end reason/tick counts for completed runs, and structured `error` records (`{code, message}`) for failures. Manifest paths are stored relative to the batch output directory so safe artifacts do not leak local absolute paths or environment data. The runner refuses to overwrite an existing batch manifest unless `--overwrite` is passed.
